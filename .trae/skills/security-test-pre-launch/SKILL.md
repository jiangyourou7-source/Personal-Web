# 安全测试上线检查清单

本技能基于火山引擎安全测试上线检查规范，提供完整的上线前安全测试流程和检查清单。适用于所有需要上线的 Web 应用、API 服务、小程序等项目。

## 使用场景

当用户进行以下操作时，应调用此技能：
- 项目即将上线，需要进行安全测试
- 用户询问"上线前需要做哪些安全检查"
- 用户要求进行安全评估或安全审计
- 需要生成安全测试报告
- 代码审查中的安全检查环节

---

## 一、安全测试流程总览

```
需求分析 → 测试计划 → 环境准备 → 测试执行 → 缺陷修复 → 回归测试 → 测试报告 → 上线审批
```

### 1.1 需求分析与计划
- 明确安全测试目标和范围
- 确定测试方法和工具
- 制定测试时间表
- 确定参与人员和职责

### 1.2 环境准备
- 搭建与生产环境相似的测试环境
- 准备测试数据（含正常和异常数据）
- 确认测试工具已就绪
- 确认测试账号已创建

---

## 二、安全测试检查清单

### 2.1 身份认证与访问控制

#### 身份认证
- [ ] 登录机制是否安全（防暴力破解）
- [ ] 是否启用多因素认证（MFA）
- [ ] 密码策略是否足够强（长度、复杂度、过期时间）
- [ ] 会话管理是否安全（超时、Token刷新）
- [ ] 登录失败是否有限制机制
- [ ] 是否存在默认账号/密码
- [ ] 密码存储是否使用安全哈希算法（如 bcrypt）

#### 权限管理
- [ ] 是否遵循最小权限原则
- [ ] 垂直权限测试（不同角色间越权）
- [ ] 水平权限测试（同角色用户间越权）
- [ ] API 接口权限控制是否完整
- [ ] 管理后台访问控制是否严格
- [ ] 是否存在未授权访问的接口

#### 测试方法
```
垂直权限测试：
- 用户A登录后，记录所有XHR请求
- 用户B（低权限）尝试访问A的请求
- 若可正常操作，则存在越权问题

水平权限测试：
- 使用两个同权限测试账号
- 修改请求中的用户标识（如userId）
- 检查是否可访问其他用户数据
```

---

### 2.2 注入攻击防护

#### SQL 注入
- [ ] 所有数据库查询是否使用参数化查询
- [ ] 是否对用户输入进行了过滤
- [ ] 是否使用 ORM 框架
- [ ] 数据库错误信息是否对外暴露
- [ ] 是否进行了 SQL 注入扫描

#### XSS（跨站脚本攻击）
- [ ] 是否对输出内容进行转义
- [ ] 是否设置了 CSP（内容安全策略）
- [ ] HttpOnly Cookie 是否启用
- [ ] 用户输入框是否进行了过滤
- [ ] 富文本编辑器是否进行了 XSS 过滤

#### 命令注入/代码注入
- [ ] 是否存在系统命令执行接口
- [ ] 用户输入是否被拼接成命令
- [ ] 是否使用了安全的 API 替代方案

#### SSRF（服务端请求伪造）
- [ ] 服务端代理请求是否限制了目标地址
- [ ] 是否禁止访问内网地址
- [ ] 是否设置了请求白名单

#### XXE（XML 实体注入）
- [ ] XML 解析是否禁用了外部实体
- [ ] 上传的 XML 文件是否进行了校验

#### 反序列化漏洞
- [ ] 是否存在用户可控的反序列化操作
- [ ] 是否使用了安全的序列化库

---

### 2.3 接口安全

#### 参数校验
- [ ] 参数是否为空校验
- [ ] 参数类型校验
- [ ] 参数长度校验
- [ ] 枚举值校验
- [ ] 数据范围校验
- [ ] 文件上传类型和大小校验

#### 返回值安全
- [ ] 是否统一封装返回值
- [ ] 是否隐藏了技术细节（如堆栈信息）
- [ ] 是否过滤了敏感字段
- [ ] 错误信息是否对用户友好

#### 接口防护
- [ ] 是否配置了限流/熔断
- [ ] 是否有防重放攻击机制
- [ ] 是否有防篡改机制（签名验证）
- [ ] CORS 配置是否正确
- [ ] HTTPS 是否强制使用
- [ ] 敏感接口是否有验证码

---

### 2.4 数据安全

#### 数据传输
- [ ] 全站是否强制使用 HTTPS
- [ ] TLS 版本是否为 1.2 及以上
- [ ] 是否禁用了弱加密套件
- [ ] 敏感数据传输是否加密

#### 数据存储
- [ ] 敏感数据是否加密存储
- [ ] 密码是否使用安全哈希算法
- [ ] 敏感配置是否硬编码（应使用环境变量）
- [ ] 日志中是否记录了敏感信息
- [ ] 数据库连接信息是否安全存储

#### 数据脱敏
- [ ] 手机号是否脱敏显示
- [ ] 身份证号是否脱敏显示
- [ ] 银行卡号是否脱敏显示
- [ ] 邮箱地址是否脱敏显示

#### 数据备份
- [ ] 是否有定期备份策略
- [ ] 备份数据是否加密
- [ ] 是否测试过数据恢复流程

---

### 2.5 配置安全

#### 服务器配置
- [ ] 是否关闭了不必要的端口
- [ ] 是否移除了默认页面
- [ ] 错误页面是否自定义
- [ ] 目录列表是否禁用
- [ ] Server Header 是否隐藏
- [ ] 是否配置了安全响应头

#### 应用配置
- [ ] 是否关闭了调试模式
- [ ] 是否移除了测试代码/页面
- [ ] 管理后台路径是否修改
- [ ] API 文档是否对外暴露
- [ ] 是否存在硬编码的密钥/密码

#### 安全响应头检查清单
```
Content-Security-Policy: default-src 'self'
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
X-XSS-Protection: 1; mode=block
Strict-Transport-Security: max-age=31536000; includeSubDomains
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
```

---

### 2.6 依赖安全

- [ ] 是否检查了第三方库的已知漏洞
- [ ] 依赖版本是否为最新稳定版
- [ ] 是否使用了安全扫描工具（如 npm audit、Snyk）
- [ ] 是否有依赖更新机制
- [ ] Docker 基础镜像是否安全

---

### 2.7 日志与监控

- [ ] 关键操作是否有日志记录
- [ ] 日志是否包含足够的信息（时间、IP、用户、操作）
- [ ] 是否配置了异常告警
- [ ] 日志是否进行了脱敏处理
- [ ] 日志存储是否安全
- [ ] 是否有实时监控和告警机制

---

### 2.8 业务逻辑安全

- [ ] 是否存在逻辑漏洞（如支付金额篡改）
- [ ] 关键操作是否有二次确认
- [ ] 是否有防刷机制
- [ ] 验证码是否有效
- [ ] 短信/邮件发送是否有限制
- [ ] 订单/交易是否有幂等性校验
- [ ] 关键业务是否有审计日志

---

### 2.9 合规检查

- [ ] 隐私政策是否完善
- [ ] 用户协议是否合规
- [ ] 数据收集是否获得了用户同意
- [ ] 是否符合等保要求
- [ ] 数据存储区域是否满足合规要求
- [ ] Cookie 使用是否合规（同意机制）

## 三、安全测试工具推荐

### 3.1 自动化扫描工具
| 工具 | 用途 | 说明 |
|------|------|------|
| OWASP ZAP | Web 应用扫描 | 开源，适合入门 |
| Burp Suite | 渗透测试 | 专业级工具 |
| Nmap | 端口扫描 | 网络探测 |
| SQLMap | SQL 注入检测 | 自动化注入测试 |
| SonarQube | 代码审计 | 静态代码分析 |
| npm audit | 依赖漏洞检查 | Node.js 项目 |
| Snyk | 依赖漏洞检查 | 多语言支持 |

### 3.2 手动测试检查
- 使用浏览器开发者工具检查网络请求
- 检查 LocalStorage/SessionStorage 中的敏感数据
- 检查 Cookie 的安全属性
- 检查 HTTP 响应头

## 四、缺陷处理流程

### 4.1 漏洞等级划分
| 等级 | 描述 | 处理要求 |
|------|------|----------|
| **严重** | 可导致系统被完全控制、大规模数据泄露 | 必须修复后方可上线 |
| **高危** | 可导致部分数据泄露、权限提升 | 必须修复后方可上线 |
| **中危** | 存在安全隐患，需要特定条件才能利用 | 建议修复，制定修复计划 |
| **低危** | 安全风险较低，影响有限 | 可安排后续版本修复 |

### 4.2 修复要求
- **严重/高危漏洞**：修复后必须进行安全复测，通过后方可上线
- **中危漏洞**：制定修复计划，明确修复时间
- **低危漏洞**：记录备案，安排后续版本修复
- 若存在高危漏洞但必须按时上线，需与安全人员协商临时措施

---

## 五、上线审批检查

### 5.1 安全测试报告内容
- 测试范围和方法
- 测试工具和环境
- 发现的漏洞清单及等级
- 漏洞修复情况
- 遗留风险说明
- 测试结论和建议

### 5.2 上线前最终确认
- [ ] 所有严重/高危漏洞已修复并复测通过
- [ ] 安全测试报告已提交并审批
- [ ] 生产环境安全配置已确认
- [ ] 监控和告警已配置
- [ ] 应急响应预案已就绪
- [ ] 回滚方案已准备

---

## 六、上线后安全运维

### 6.1 持续监控
- 实时监控系统运行状态
- 监控异常访问行为
- 定期检查安全日志
- 关注安全漏洞情报

### 6.2 定期检查
- 定期进行安全扫描
- 定期更新依赖组件
- 定期审查权限配置
- 定期进行渗透测试

### 6.3 应急响应
- 建立应急响应流程
- 明确应急联系人
- 准备应急处理预案
- 定期进行应急演练

## 使用示例

当用户说"我要上线了，帮我检查一下安全"时，按以下步骤执行：

1. **了解项目类型**：确认是 Web 应用、API 服务还是其他类型
2. **检查代码安全**：
   - 搜索是否存在硬编码的密钥/密码
   - 检查是否有调试代码残留
   - 检查依赖是否有已知漏洞
3. **检查配置安全**：
   - 检查安全响应头配置
   - 检查 HTTPS 配置
   - 检查 CORS 配置
4. **检查数据安全**：
   - 检查敏感数据处理
   - 检查日志是否记录敏感信息
5. **生成检查报告**：汇总发现的问题和建议

### 快速检查命令

```bash
# Node.js 项目依赖漏洞检查
npm audit

# 检查是否使用了 HTTP（应使用 HTTPS）
grep -r "http://" --include="*.ts" --include="*.js" --include="*.tsx" --include="*.jsx" src/

# 检查硬编码的密钥
grep -r "password\|secret\|key\|token" --include="*.ts" --include="*.js" --include="*.env" .

# 检查调试代码
grep -r "console.log\|debugger\|TODO\|FIXME" --include="*.ts" --include="*.js" src/
```
